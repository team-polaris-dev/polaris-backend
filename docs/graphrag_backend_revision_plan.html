POLARIS GraphRAG Plan 진단 외부 사례 코드 비교 목표 구조 수정 순서 검증 출처
작성일 2026-06-14 · backend GraphRAG revision plan

자유 질문을 받기 위한 GraphRAG 백엔드 수정 계획

현재 POLARIS는 데이터 적재와 각 DB 검색기는 꽤 갖춰졌지만, 질문을 읽고 어떤 DB에 어떤 형태로 요청할지 정하는 계층이 약합니다. 로컬 LLM 전제의 템플릿 중심 구조에서, Apimaker를 이용한 질문 분석-계획-검증 루프 중심 구조로 바꾸는 것이 핵심입니다.

결론

사용자의 판단처럼 병목은 대체로 검색 이후보다 검색 이전입니다. 다만 검색 이후도 완전히 안전하지는 않습니다. 현재 합성 단계는 검색 결과를 문자열로 합치기 때문에 점수, 출처, self-check, 실행 쿼리 같은 근거 메타가 답변까지 살아남지 못합니다.

가장 먼저 고칠 것

QueryPlanner structured JSON tool-specific request retry/repair loop

`ctx` 뒤에 계획 노드를 넣고, `rdb/vec/graph`를 무조건 병렬 실행하지 말고 계획된 도구만 실행하도록 바꿉니다.

01

현재 진단

핵심 문제: 지금 상위 LangGraph는 `direct` 또는 `ctx`만 고른 뒤, 공시 질문이면 `rdb`, `vec`, `graph`를 모두 같은 재구성 질문으로 실행합니다. 즉 자유 질문을 도구별 요청으로 분해하는 계획자가 없습니다.

1. 상위 라우터가 너무 얕음

`nodes/router.py`는 `ctx/direct`만 반환합니다. "재무 비교 + 공급망 리스크 + 출처"처럼 복합 질의가 들어와도 RDB, Vector, Graph에 서로 다른 하위 질의를 주지 못합니다.

파일: polaris-backend/nodes/router.py:31-57

2. search_plan은 선언만 있음

`AgentState`와 채팅 로그 테이블에는 `search_plan`이 있지만 실제 생성 노드가 없습니다. 관리자 통계는 도구 사용량을 읽을 준비가 되어 있는데, 상태에는 값이 채워지지 않습니다.

파일: core/state.py:18-20, main.py:100-106

3. reflection 루프가 꺼져 있음

충분성 검증이 실패해도 현재 코드는 개발 단계 로직으로 바로 `is_sufficient=True`를 반환합니다. 자유 질문에서 가장 필요한 "질문 재작성 후 재검색" 루프가 사실상 동작하지 않습니다.

파일: nodes/rag.py:277-283
현재 구현 좋은 점 자유 질문에서 막히는 이유
GraphAgentRunner 12의도, route, stage, 템플릿, Text-to-Cypher, guard, self_check가 이미 있음 상위 파이프라인이 이 결과를 검색 계획으로 재사용하지 않고, wrapper가 메타를 대부분 버림
rdb_search_node Apimaker 기반 NL-to-SQL과 SELECT-only guard가 있음 질문 전체를 바로 SQL로 바꾸므로 복합 질문의 RDB 부분만 분리하지 못함
vector_search_node Dense + BM25 + RRF와 회사/연도 필터 추출이 있음 검색어, 필터, 문서 타입, 답변 목적이 계획에서 내려오지 않음
synthesizer_node 3개 DB 결과를 하나로 묶는 최소 구현이 있음 구조화 evidence가 아니라 문자열이라 점수, 출처, 실행 쿼리, 누락 사유가 손실됨
02

외부 사례 분석

아래 캡처는 2026-06-14에 공식 문서 페이지를 직접 캡처한 것입니다. 비공식 블로그보다 문서와 공개 코드가 함께 있는 사례를 우선했습니다.

Microsoft GraphRAG

원문에서 강조하는 구조는 텍스트를 TextUnit으로 나누고, 엔티티/관계/커뮤니티 요약을 만든 뒤 질의 시 Global, Local, DRIFT, Basic Search 중 맞는 검색 모드를 쓰는 방식입니다.

community summaries local search global search
소스: GraphRAG docs, Global Search, Local Search

Neo4j GraphRAG Text2Cypher

Neo4j의 Text2CypherRetriever는 자연어 질문을 LLM으로 Cypher로 바꾸고, 실행 결과 레코드를 다시 답변 컨텍스트로 넣는 분리 구조입니다. Cypher 생성 LLM과 답변 LLM을 나눌 수 있습니다.

schema prompt Text2Cypher result formatter
소스: Neo4j GraphRAG RAG Guide, Text2Cypher source docs

LlamaIndex PropertyGraphIndex

LlamaIndex는 chunk마다 여러 `kg_extractors`를 적용해 엔티티/관계를 노드 메타로 붙입니다. 특히 SchemaLLMPathExtractor는 허용 엔티티/관계와 연결 규칙을 검증합니다.

kg_extractors schema validation ingestion-first
소스: LlamaIndex Property Graph guide

LangGraph Agentic RAG

LangGraph 예시는 검색 도구 사용 여부 결정, 문서 관련성 평가, 질문 재작성, 최종 생성 노드를 분리합니다. 자유 질문 대응에는 이 "grade and rewrite" 루프가 중요합니다.

tool decision grade documents rewrite question
소스: LangGraph custom RAG agent
POLARIS에 바로 가져올 점: Microsoft의 검색 모드 분리, Neo4j의 자연어-Cypher 분리, LlamaIndex의 schema-driven 추출 관점, LangGraph의 질문 재작성 루프를 모두 합치되, POLARIS의 장점인 DART 정형 데이터와 Cypher guard는 그대로 유지합니다.

Microsoft Global Search가 주는 힌트

커뮤니티 리포트를 바로 최종 답변에 넣는 것이 아니라, 중간 응답을 만들고 중요한 포인트만 다시 줄이는 map/reduce식 흐름이 핵심입니다. POLARIS의 `community_cards`도 이 방향으로 확장하면 "반도체 생태계 전체를 요약해줘" 같은 질문을 관계 1개짜리 템플릿으로 억지 처리하지 않아도 됩니다.

캡처 소스: Microsoft GraphRAG Global Search
03

현재 코드와 사례 코드 비교

영역 외부 사례의 패턴 POLARIS 현재 코드 수정 방향
검색 모드 선택 Microsoft GraphRAG는 Global, Local, DRIFT, Basic을 질의 성격에 따라 나눔 router_node는 `ctx/direct`만 선택. 이후 `rdb/vec/graph` 모두 실행 Apimaker 기반 `QueryPlanner`가 tool과 mode를 구조화 JSON으로 선택
Text-to-Cypher Neo4j는 schema/examples/custom prompt를 넣어 Cypher를 만들고 record를 컨텍스트화 TextToCypher.generate와 validate_cypher는 이미 유사하게 구현 상위 plan의 entities/slots/relations를 runner에 주입하고 결과 메타를 보존
그래프 추출 LlamaIndex는 `kg_extractors`와 strict schema extractor로 추출 규칙을 명시 POLARIS는 `pipeline_scripts/graph/extract_*.py`가 회사별/문서별 스크립트로 흩어져 있음 질문 단계보다 후순위지만, 공급망/제품/기술 추출은 schema registry로 통일
자기 수정 루프 LangGraph 예시는 grade 실패 시 질문을 rewrite하고 다시 검색 reflection_node는 실패해도 현재 바로 통과 처리 coverage 실패 시 `repair_plan`으로 재검색. 2회 제한 후 부족함을 답변에 표시
관측 가능성 retriever 결과와 query metadata를 분리해 디버그 가능 graph_agent.py 주석상 runner의 13키 중 대부분을 버림 `ToolTrace`, `graph_debug`, `cypher_executed`, `self_check`를 상태와 응답 옵션에 보존

POLARIS의 이미 좋은 부분

  • Apimaker in-process LLM 어댑터가 이미 들어와 있음
  • GraphAgentRunner는 의도, route, stage, template, Text-to-Cypher, self_check를 갖춤
  • Cypher guard가 write/admin 차단, 라벨/관계 whitelist, LIMIT, EXPLAIN dry-run을 수행
  • Vector는 Dense + BM25 + RRF를 이미 사용
  • MariaDB는 SELECT-only 실행기와 큐레이션된 schema prompt를 갖춤

현재 가장 큰 누락

사용자 자유 질문
  -> 대화 맥락 재구성
  -> [없음] 도구별 검색 계획
  -> 같은 질문을 rdb/vec/graph 모두에 투입
  -> 문자열 합성
  -> 부족해도 통과

필요한 구조
  -> QueryPlanner JSON
  -> 도구별 sub-query / filters / slots
  -> selective 또는 parallel tool execution
  -> structured EvidenceBundle
  -> coverage check
  -> repair or final answer
04

목표 구조

QueryPlanner 출력 스키마 초안

{
  "needs_retrieval": true,
  "answer_shape": "comparison | list | number | explanation",
  "tasks": [
    {
      "tool": "rdb",
      "purpose": "2024 revenue comparison",
      "query": "삼성전자와 SK하이닉스 2024년 매출액",
      "slots": {"year": 2024, "account_id": "ifrs-full_Revenue"},
      "entities": ["00126380", "00164779"]
    },
    {
      "tool": "graph",
      "mode": "local",
      "intent": "supply_chain",
      "relations": ["supplies"],
      "entities": ["00126380", "00164779"],
      "max_hops": 2
    }
  ],
  "fallback": {"if_no_entity": "ask_clarification"}
}

핵심 설계 원칙

  • LLM은 planner로 쓰고, 숫자와 관계는 DB가 결정한다.
  • 템플릿 우선, Text-to-Cypher 보조를 유지한다.
  • 검색 결과는 문자열이 아니라 Evidence 객체로 합친다.
  • 부족하면 rewrite/repair한다. 부족함을 숨기고 답하지 않는다.
  • 관측 메타를 버리지 않는다. 실행 SQL/Cypher, self_check, source, score를 남긴다.
05

수정 순서

0

상태와 로그 확장

`AgentState`에 `query_plan`, `tool_traces`, `evidence_bundle`, `graph_debug`, `planner_errors`를 추가합니다. `ChatResponse`는 기본 응답은 유지하되 debug 옵션으로 plan/trace를 받을 수 있게 합니다.

대상: core/state.py, main.py, services/chat_logging.py
1

Apimaker 기반 QueryPlanner 추가

`nodes/planner.py`를 새로 만들고, `reconstructed_query + 대화 기록 + DB 역할 설명 + graph intent catalog`를 입력해 엄격한 JSON plan을 생성합니다. 실패 시 규칙 fallback은 가능하지만 fallback 여부를 trace에 남깁니다.

대상: nodes/planner.py, prompts/query_plan.md
2

core/graph.py를 계획 기반 조건부 실행으로 변경

`ctx -> rdb/vec/graph` 전체 병렬 실행을 `ctx -> plan -> planned tools`로 바꿉니다. 복합 질문이면 선택된 도구만 병렬 실행하고, 단일 질문이면 하나만 실행합니다.

대상: core/graph.py:64-71 교체
3

도구별 adapter가 plan을 소비하도록 변경

RDB는 `tasks[].query/slots/entities`를 기준으로 SQL을 만들고, Vector는 `semantic_query + filters`를 받습니다. Graph는 `GraphAgentRunner.run({"query", "intent", "entities", "slots", "requested_relations", "max_hops"})` 형태로 planner 결과를 직접 주입합니다.

대상: nodes/rag.py, graphrag/langgraph_app/nodes/graph_agent.py
4

EvidenceBuilder로 합성 계층 교체

`synthesized_info` 문자열은 최종 prompt용 렌더링 결과로만 남기고, 내부 상태는 `{items, source_map, scores, missing, executed_queries}` 형태로 유지합니다. 이렇게 해야 답변 근거와 누락 사유를 정확히 제어할 수 있습니다.

대상: nodes/rag.py:synthesizer_node
5

Reflection을 coverage checker + repair loop로 복구

`current_retry >= 0` 강제 통과 조건을 제거합니다. coverage가 낮으면 planner로 되돌아가 `prior_missing`과 `tool_errors`를 넣고 1-2회 재계획합니다.

대상: nodes/rag.py:250-286
6

프론트/관리자에서 plan trace 확인

사용자는 평소에는 답변만 보고, 개발/관리자 모드에서는 어떤 도구를 왜 썼는지, 어떤 SQL/Cypher가 실행됐는지, 어느 단계에서 누락됐는지 확인할 수 있게 합니다.

대상: polaris-frontend Chatbot/Admin Analytics
주의: `GraphAgentRunner`를 버릴 필요는 없습니다. 오히려 가장 완성도가 높은 부분입니다. 수정의 중심은 GraphAgent 자체가 아니라, 그 앞단의 planner와 뒷단의 evidence/coverage입니다.
06

검증 기준

Plan validity

자유 질문 50개에 대해 planner JSON이 schema를 통과하고, 필요한 도구를 빠뜨리지 않는지 봅니다. 복합 질문은 `tasks.length > 1`이 자연스럽게 나와야 합니다.

Entity/slot F1

회사명, corp_code, 연도, IFRS account_id, 관계 타입을 얼마나 잘 잡는지 따로 측정합니다. "DB 정확도"라는 말은 이 단계와 실행 결과 정확도를 분리해서 봐야 합니다.

Cypher/SQL safety

generated query가 guard를 통과하는 비율, guard 실패 후 retry 성공률, 실행 시간, 빈 결과의 원인 분류를 로그로 남깁니다.

Context recall

정답 근거가 top-k evidence 안에 들어오는지 측정합니다. Graph는 `graph_path_validity`, Vector는 chunk recall, RDB는 row match를 분리합니다.

Grounded answer

답변이 evidence에 없는 숫자/관계를 만들지 않는지 봅니다. 부족한 경우 "자료 없음"을 말하는 것도 성공 케이스로 둡니다.

Latency budget

planner 1회, 도구 실행, coverage check, repair 재시도별 시간을 나눠 기록합니다. 모든 질문에 모든 DB를 치는 현재 방식보다 평균 지연은 줄어야 합니다.

테스트 질문 세트 예시

1. "삼성전자와 SK하이닉스 중 2024년 매출이 더 크고, 공급망 관계가 더 많이 잡힌 곳은?"
2. "한미반도체의 주요 고객사와 그 근거 공시를 같이 보여줘."
3. "삼성 계열 반도체 생태계를 한 문단으로 요약하고 핵심 관계 5개만 뽑아줘."
4. "작년에 매출이 늘어난 회사 중 공급망 관계가 있는 회사를 찾아줘."
5. "그 회사 자회사 중 재무 데이터가 있는 곳만 표로 정리해줘."
