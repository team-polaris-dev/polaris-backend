여덟 단계짜리 GraphRAG를 두 가지 일만 하는 검색기로 다시 짠 이야기 — 그리고 그 검색이 기대는 단어집을 만든 과정.

Neo4j FULLTEXT·Cypher 다중-hop·APOC·단어집 4종

들어가며

POLARIS는 기업 공시 데이터를 분석하는 시스템이다. 사용자가 "삼성전자 대표이사가 누구야?", "SK하이닉스 공급망 알려줘" 같은 질문을 던지면, 시스템은 세 갈래 검색(정형 데이터, 문서 본문, 관계망)을 동시에 돌려 답을 만든다. 이 글은 그중 관계망 검색, 즉 GraphRAG를 어떻게 다시 짰는지에 대한 기록이다.

기존 구현은 손이 많이 갔다. 질문이 들어오면 먼저 의도를 분류하고, 분류 결과에 맞는 Cypher 템플릿을 고르고, 안 맞으면 LLM에게 Cypher를 새로 짜게 시키고, 그 Cypher가 위험하지 않은지 검사한다. 단계가 여덟 개나 됐다. 단계가 많다는 건 고장 날 지점도 많다는 뜻이다.

그래서 방향을 바꿨다. 앞단에서 단어집을 보고 질문을 다듬어 넘겨주기로 했으니, 그래프 검색은 두 가지만 잘하면 된다. 질문에서 회사를 정확히 찾아내는 것, 그리고 그 회사를 중심으로 관계망을 펼치는 것.

STEP 1질문에서 회사 찾기

가장 먼저 할 일은 질문 속 단어를 그래프의 노드와 잇는 것이다. 사용자는 "삼성전자"라고 쓰지만 데이터베이스에는 "삼성전자(주)"로 적혀 있다. 이 간극을 메우는 게 첫 관문이다.

여기서 Neo4j가 기본으로 제공하는 전문 검색(FULLTEXT INDEX)을 썼다. 회사명·인물명·제품명을 미리 색인해두면, 사용자가 정확한 표기를 모르고 일부만 쳐도 가장 비슷한 노드를 찾아준다. 한국어는 두 글자씩 잘라 색인하는 방식(cjk)을 골랐다. "삼성전자"라고만 쳐도 "삼성전자(주)"가 맨 위로 올라온다.

-- 한국어 회사·인물·제품·기술을 cjk 분석기로 색인
CREATE FULLTEXT INDEX entity_fulltext IF NOT EXISTS
FOR (n:Organization|Person|Product|Technology)
ON EACH [n.name, n.er_name]
OPTIONS {indexConfig:{ `fulltext.analyzer`: 'cjk' }};

-- 질의 시: 부분 표기로도 가장 가까운 노드를 회수
CALL db.index.fulltext.queryNodes('entity_fulltext', '삼성전자')
  YIELD node, score RETURN node.name, score ORDER BY score DESC;
-- → 삼성전자(주)  score 13.4   (맨 위)
벡터는 왜 안 썼나. 별도의 임베딩이나 벡터 검색은 여기 쓰지 않았다. 의미 기반 검색은 이미 시스템의 다른 갈래(Qdrant)가 맡고 있어서, 그래프 쪽까지 벡터를 또 얹으면 데이터를 두 군데서 맞춰야 하는 부담만 생긴다.

STEP 2관계망 펼치기

회사를 찾았으면 그 회사를 기점으로 관계를 따라간다. 임원이 누구인지, 주주는 누구인지, 어떤 자회사를 거느리는지, 누구에게 납품하는지.

여기서 한 가지 고민이 있었다. 관계마다 따라가야 할 깊이가 다르다는 점이다. 임원 관계는 한 칸이면 충분하다. 사람과 회사가 바로 이어지니까. 하지만 지배구조는 다르다. 자회사의 자회사의 자회사까지 거슬러 올라가야 그림이 완성된다. 공급망도 마찬가지로 1차, 2차, 3차 협력사까지 이어진다.

그래서 관계 종류마다 펼치는 깊이를 다르게 줬다. 임원은 한 칸, 지배구조는 다섯 칸, 공급망은 세 칸. 한 가지 깊이로 모든 관계를 훑으면, 얕으면 놓치고 깊으면 엉뚱한 노드까지 끌려와 답이 지저분해진다.

FIG 2 — 관계 종류마다 펼치는 깊이가 다르다 (초록 = 공급망, 방향 구분)

공급망에서는 방향도 살렸다. A가 B에게 납품하는 것과 B가 A에게 납품받는 것은 분석에서 의미가 정반대다. 그래서 "이 회사가 공급하는 곳"과 "이 회사에 공급하는 곳"을 나눠서 가져온다.

// 공급망 — 방향을 나눠 buyer / supplier 따로 회수
MATCH (o:Organization {corp_code:$code})
CALL (o) {
  OPTIONAL MATCH p=(o)-[:SUPPLIES_TO*1..3]->(buyer:Organization)
  RETURN collect({name:buyer.name, tier:length(p), role:'buyer'}) AS buyers }
CALL (o) {
  OPTIONAL MATCH p=(supplier:Organization)-[:SUPPLIES_TO*1..3]->(o)
  RETURN collect({name:supplier.name, tier:length(p), role:'supplier'}) AS suppliers }
RETURN buyers, suppliers;
왜 CALL 서브쿼리인가. 임원·주주·투자·공급을 한 MATCH에 이어 붙이면 결과가 카르테시안 곱으로 폭발한다. 각 관계를 서브쿼리로 격리해 곱셈을 끊었다.

STEP 3빈손으로 돌아오지 않기

정해진 관계 패턴만 따라가면, 미처 생각 못 한 질문에서는 빈손이 될 수 있다. 이때를 대비해 안전망을 뒀다. 정해진 패턴이 아무것도 못 건지면, 찾은 회사 주변의 그래프를 통째로 한 번 떠 온다(APOC subgraphAll). 관계 종류를 따지지 않고 일단 근처를 다 끌어오는 방식이라, 예상 밖의 질문에도 무언가는 건네줄 수 있다.

안전망의 조건. fallback은 회사는 찾았는데 정해진 패턴이 0건일 때만 작동한다. 애초에 회사 자체를 못 찾았으면 시작 노드가 없으니 fallback도 돌지 않는다.

단어집 — 왜 필요했나

검색이 첫 관문에서 막히는 경우가 있었다. "SK하이닉스"라고 치면, 데이터베이스의 정식 이름은 "에스케이하이닉스(주)"라서 글자가 안 맞아 엉뚱한 노드가 잡혔다. 회사를 못 찾으면 그 뒤 관계망도 통째로 빈다.

질의매칭된 노드회수된 사실
SK하이닉스 공급망엉뚱한 노드5
에스케이하이닉스 공급망에스케이하이닉스(주)720
FIG 3 — 같은 회사인데 표기 하나로 5건 vs 720건. 첫 관문이 전부를 좌우한다

이걸 메우는 게 단어집이다. 단어집은 검색을 직접 하지 않는다. 앞단의 LLM이 사용자의 구어를 데이터베이스 용어로 다듬을 때 참고하는 사전이다. "돈 얼마 벌었어?"를 "매출액"으로, "사장"을 "대표이사"로, "연간 실적"을 "사업보고서"로 바꿔주는 길잡이다.

단어집 — 어떻게 만들었나

단어집에 담을 건 네 종류다. 회사·인물·제품 같은 이름, 매출·영업이익 같은 재무 용어, 자회사·주주 같은 관계어, 그리고 연결·별도 같은 코드값.

① 엔티티 이름
organization · person · product
삼성전자(주), 이재용, HBM …
▲ 그래프에서 자동 추출 · degree 내림차순
② 재무 계정
fin_accounts
매출 → ifrs-full_Revenue
▲ 손으로 매핑 (DB에 한글 0개)
③ 관계 술어
relation_predicates
자회사 → IS_SUBSIDIARY_OF
▲ 손으로 매핑 · 고정 어휘
④ 코드값
reprt_codes · fs_div
연결 → CFS, 사업보고서 → 11011
▲ 손으로 매핑 · 고정 어휘

이름은 그래프에서 자동으로 뽑았다. 많이 등장하는 순서로 정렬했는데, LLM이 긴 목록의 중간보다 앞쪽을 더 잘 본다는 점을 노린 배치다.

나머지 세 종류는 데이터베이스에 한글이 아예 없었다. 재무 계정은 전부 영문 IFRS 코드뿐이라, "매출"이라는 단어를 이을 한글이 한 글자도 없었다. 그래서 자주 쓰는 계정 서른 개의 한글 매핑을 직접 만들었다. 관계어와 코드값도 같은 방식으로 손으로 채웠다. 이쪽은 거의 변하지 않는 고정된 어휘라, 한 번 만들어두면 끝이다.

처음에는 이름은 데이터 파일에, 고정 어휘는 코드에 나눠 적었다. 그런데 단어집이 두 군데로 쪼개지는 게 어색했다. 그래서 전부 하나의 JSON 파일로 합치고, 코드는 그 파일을 읽기만 하게 정리했다. 팀원은 이제 파일 하나만 받으면 된다.

// vocab.json — 정적(사람 관리) + 동적(DB 추출)을 한 파일에
{
  "schema_summary": "...",           // 스키마 설명
  "fin_accounts": { "ifrs-full_Revenue": ["매출액","매출","수익"] },
  "relation_predicates": { "IS_SUBSIDIARY_OF": ["자회사","종속회사"] },
  "reprt_codes": { "11011": ["사업보고서","연간"] },
  "organization": [ {"id":"00126380", "name":"삼성전자(주)"} ]  // 자동
}

출력 — 기존 코드를 건드리지 않기

검색 결과는 하나의 목록으로 낸다. 회사든 인물이든 관계든, 전부 같은 형식의 "사실" 한 줄로 표현한다. 다만 기존 시스템이 옛 형식을 읽고 있어서, 새 형식과 옛 형식을 동시에 내보낸다. 덕분에 다른 사람이 짠 코드를 고치지 않고도 새 검색이 그대로 맞물린다.
