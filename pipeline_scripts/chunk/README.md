# 사업보고서 본문 청킹 (POLARIS)

DART 공시 원본은 DART XML(dart4.xsd) 형식이다. POLARIS는 **정기보고서(사업·반기·분기)** 만 원본을 받아 본문을 청킹하고, 나머지 공시는 구조화 API(JSON)로 적재한다(청킹 안 함). 청크는 의미검색(Qdrant bge-m3 1024d)과 관계추출(Claude)에 쓰인다.

---

## 1. 무엇을 청킹하나

기준은 사업보고서이며, 3사(삼성전자·SK하이닉스·한미반도체)의 챕터 구조가 동일하다.

**원칙:** 숫자·지분·임원 같은 **정형 데이터는 DART API(JSON)가 정확**하므로 청킹하지 않고 그래프로 직행한다(LLM 환각 방어). **서술·주석은 API에 없어서** 본문 청킹이 유일한 소스다.

| 구분 | 챕터 | 처리 |
|---|---|---|
| 청킹 대상 (서술·주석) | II.사업의 내용 / III의 3.연결재무제표 주석·5.재무제표 주석(특수관계자 거래 등) / IV.경영진단(MD&A) / V.감사인의 감사의견 본문 / IX.계열회사 관계 서술 / X.대주주 거래내용 / XI.기타 투자자보호 / 감사보고서(연결 포함)의 핵심감사사항 | 본문 청킹 → Qdrant + Claude |
| 청킹 스킵 (정형 SSOT) | I.회사개요(자본금·자기주식) / III의 1.요약재무·2.연결재무제표·4.재무제표(본체)·6.배당·7·8(자금·차입) / VI.이사회 / VII.주주 / VIII.임원·직원·보수 / XII.상세표 | DART API(JSON) → 재무·지분·임원 그래프 직행 |

## 2. 산문 청킹 (`chunk_type = text_micro`)

- RecursiveCharacterTextSplitter, **800자 / 80자(10%) 오버랩** (문단 → 문장 → 문자 순 재귀 분할).
- 각 청크 앞에 **프리픽스 헤딩**을 붙인다(Contextual Retrieval). 형식: `[회사명 · 문서(YYYY.MM) · 섹션경로]`. 프리픽스는 `embedding_text`에만 넣는다.

예시:

```
[삼성전자 · 사업보고서(2024.12) · II.사업의 내용 > 1.사업의 개요]
당사는 본사를 거점으로 한 DX부문과 DS부문으로 구성되어 있으며 ...
```

한국어 공시는 한 문단이 길고 조사로 의미가 이어져, 800/80이 검색 정밀도(과도하게 잘리지 않음)와 충전율(~89%)의 균형점이다.

---

## 3. 표 청킹 (`chunk_type = table_nl`)

- 표는 **행 리스트로 펼쳐 자연어화**한다. 행마다 `헤더=값`. rowspan/colspan 병합셀은 값을 펼치고, 2줄짜리 헤더는 하나로 합친다.
- **표 1개 = 1청크**가 기본(행 단위로 안 쪼갬). 단위 캡션 `(단위: 백만원)`은 머리에 보존한다.
- 셀 수가 매우 큰 표(약 120 초과)만 **행그룹으로 분할**하고, 이때 헤더행을 매 조각에 반복한다. 레이아웃용 단일셀 표(약 60%)는 스킵한다.

예시 (특수관계자/재무 주석 표 → table_nl):

```
[삼성전자 · 사업보고서(2024.12) · III.재무 > 5.재무제표 주석 > 특수관계자]
(단위: 백만원)
기초 | 당기=9,727,674; 전기=8,853,712
배당금 | 당기=(1,022,540); 전기=(43,646)
```

## 4. 노이즈 — 무엇을 제거/보존하나

| 처리 | 대상 | 이유 |
|---|---|---|
| 제거 | 전각공백(`　`)→일반공백 / 연속공백→1칸 / 빈 줄 3연속+ 축약 / 각주마커(`주N)`) / 불릿·주의기호(`※ □ ○`) / 표참조마커(`<표N>`·`[표N]`) | 의미 없는 서식 노이즈, 토큰 낭비·검색 방해 |
| 보존 | `△`(재무 음수) / `㈜`(회사명) / `(단위:…)` / 한자(한글 병기 의미) / `「」`(법령 인용) | 제거하면 숫자·고유명사·법적 의미가 손상됨 |

## 5. XML 처리 주의

DART XML에는 **이스케이프 안 된 `&`** 가 섞여 있어, 엄격한 XML 파서(ElementTree / lxml strict)는 그 지점에서 멈춘다. (브라우저로 열면 잘린 것처럼 보이지만 파일 자체는 멀쩡하다.) → **정규식 또는 lxml(recover)** 로 파싱한다.

## 6. 산출물

`chunk/output/{회사}_{접수번호}.jsonl` — 청크 1줄 = 1개.

| 필드 | 설명 |
|---|---|
| `chunk_id` | 16hex |
| `corp_code` | 회사 코드(8) |
| `rcept_no` | 접수번호(14) |
| `chunk_type` | `text_micro` 또는 `table_nl` |
| `section_path` | 섹션 경로 (예: `II.사업의 내용 > 1.사업의 개요`) |
| `embedding_text` | 프리픽스 헤딩 포함 본문(임베딩 입력) |
| `token_count` | 토큰 수 |

## 7. 알려진 한계 (표 헤더)

표 헤더는 휴리스틱으로 검출한다(선두의 숫자비율 낮은 행 = 헤더, 최대 4단, 컬럼명은 하위 2단계만). 대부분의 표 — 매출·생산능력·환위험(2~3단)·특수관계자 거래·채권채무 — 는 정확히 변환된다.

**다만 DART 재무제표 주석의 4~6단 초심층 중첩표**(예: 공정가치 서열체계, 금융상품 분류)는 헤더-값 매핑이 불완전할 수 있다(헤더 장황 또는 일부 전치). 이는 단순 규칙으로 100% 해결이 어려운 본질적 한계이며, 헤더 깊이 캡(4)으로 과흡수를 막아 **숫자 데이터 자체는 손실되지 않는다**(매핑만 흐려짐). 해당 표는 table_nl의 소수(주석 구간 집중)다.

필요 시 후속 개선: 초심층 표만 별도 식별해 행 원본 덤프 또는 Claude 정규화로 처리.
